# Архитектура мобильного приложения «Контроль успеваемости»

Приложение построено по многослойной архитектуре с разделением на **Presentation**, **Domain** и **Data** слои, с поддержкой офлайн-режима и фоновой синхронизации.

## Цели архитектуры

- **Масштабируемость** — добавление нового функционала без ломки существующего.
- **Тестируемость** — изоляция бизнес-логики (Use Cases) для unit-тестов.
- **Поддерживаемость** — явное разделение слоёв.
- **Надёжность** — корректная работа при нестабильной сети (офлайн, повторная синхронизация).
- **Безопасность** — хранение токена (EncryptedSharedPreferences/DataStore), HTTPS, JWT.

## Направление зависимостей

```
Presentation → Domain ← Data
```

Domain не зависит от UI и от деталей реализации данных (сеть, БД).

---

## 1. Presentation Layer (Слой представления)

- **UI**: Activity, Fragment, ViewBinding.
- **ViewModel**: состояние экрана, вызов Use Cases или репозиториев, передача данных во View.
- **Поток данных**: пользовательское действие → ViewModel → Use Case / Repository → обновление состояния → перерисовка UI.

---

## 2. Domain Layer (Слой бизнес-логики)

- **Интерфейсы** (контракты для Data): например, `ISyncCatalog` — синхронизация справочников.
- **Use Cases**: атомарные сценарии.
  - `SyncCatalogUseCase(syncCatalog: ISyncCatalog)` — запускает синхронизацию каталогов (факультеты, группы, дисциплины, студенты). Используется из `SyncWorker` и при ручном обновлении в MainActivity.

Domain не зависит от Android-библиотек, что упрощает unit-тестирование.

---

## 3. Data Layer (Слой данных)

- **Репозитории**: реализация доступа к данным (Room, API).
- **Remote**: Retrofit + OkHttp, JWT в заголовках (SessionManager, интерцептор).
- **Local**: Room (сущности, DAO) — кэш и офлайн-хранилище; при необходимости — DataStore/EncryptedSharedPreferences для токена и настроек.
- **Sync**: `SyncManager` реализует `ISyncCatalog`, выполняет загрузку с сервера и сохранение в Room.

---

## Ключевые потоки данных

### Просмотр списка оценок (с синхронизацией с сервера)

1. Пользователь открывает экран оценок.
2. `GradeViewModel` при загрузке (если передан API) вызывает синхронизацию оценок студента с API: запрос `GET grades/student/{id}`, маппинг DTO → сущности Room, замена локальных данных.
3. Данные читаются из Room через репозиторий и отображаются во View.

### Синхронизация справочников

1. **При входе в приложение**: в `MainActivity.onCreate()` запускается `SyncManager.sync()` в корутине (факультеты, группы, дисциплины, студенты).
2. **В фоне**: WorkManager раз в 15 минут запускает `SyncWorker`, который использует `SyncCatalogUseCase` и тот же `SyncManager` для повторной синхронизации. При ошибке возвращается `Result.retry()` для повторной попытки.

---

## Офлайн и синхронизация

- Локальные данные хранятся в Room; при наличии сети они обновляются с сервера.
- Оценки студента подтягиваются с API при открытии экрана оценок (если пользователь авторизован и передан API в ViewModel).
- Справочники синхронизируются при старте приложения и по расписанию через WorkManager.

---

## Безопасность

- JWT и данные сессии хранятся через SessionManager (безопасное хранилище).
- Сетевые запросы по HTTPS; токен передаётся в заголовке `Authorization`.

---

## Расширение

Архитектура допускает добавление:

- новых Use Cases (например, синхронизация только оценок по расписанию);
- полнотекстового поиска и фильтров на сервере и в локальном кэше;
- очереди отложенных операций с повторными попытками (по аналогии с «Кулинарный блог»).
