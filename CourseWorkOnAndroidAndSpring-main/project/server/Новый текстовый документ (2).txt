В работе реализованы мобильный Android-клиент (Kotlin, Android Studio) с локальным кэшем (Room) и фоновой синхронизацией справочников (WorkManager), а также серверная часть на Spring Boot с REST API. На сервере используется PostgreSQL (миграции Flyway) и JWT-аутентификация. Токен на клиенте хранится в SharedPreferences (SessionManager). The work implements a mobile Android client (Kotlin, Android Studio) with local cache (Room) and background synchronization of catalogs (WorkManager), and a Spring Boot server with REST API. The server uses PostgreSQL (Flyway migrations) and JWT authentication. The client stores the token in SharedPreferences. Введение

Современные веб- и мобильные системы в сфере образования автоматизируют учёт успеваемости, управление справочными данными (факультеты, группы, дисциплины, студенты, преподаватели), взаимодействие участников (журнал оценок и посещаемости, отчётность) и формирование аналитики; при использовании разрозненных средств рост объёма данных ведёт к дублированию и снижению удобства работы. Внедрение клиент-серверной АИС обеспечивает целостность данных, ускоряет формирование отчётов и повышает масштабируемость решения.
Актуальность: Разработка АИС «Учёт успеваемости студентов» обоснована потребностью в централизованном хранении академических данных, синхронизации мобильных клиентов при нестабильной сети и получении отчётности по группам и студентам.
Цель. Создать клиент-серверную АИС для учёта успеваемости студентов, обеспечивающую удобное внесение, хранение и поиск оценок и посещаемости, безопасное взаимодействие пользователей с разграничением прав и формирование отчётов (Excel/PDF, доставка по e-mail или сохранение на устройство).
Задачи:
- Спроектировать архитектуру: сервер на Spring Boot (Java) с REST API и мобильный клиент для Android (Kotlin) с локальным хранилищем и фоновой синхронизацией.
- Реализовать CRUD для факультетов, групп, дисциплин, студентов, преподавателей, пользователей, оценок и посещаемости.
- Обеспечить аутентификацию и авторизацию (JWT), разграничение ролей (администратор, деканат, преподаватель, студент) и защиту эндпоинтов.
- Реализовать формирование и выгрузку отчётов (Excel/PDF) с возможностью отправки на e-mail или сохранения на устройство.
- Обеспечить локальное хранилище на клиенте (Room, хранение токена в SharedPreferences) и фоновую синхронизацию справочников с сервером.
- Подготовить административные и пользовательские API, подсистему отчётности; предусмотреть возможность расширения (дополнительные отчёты, уведомления).
Практическая значимость: развёртывание (Spring Boot, PostgreSQL, Flyway), журнал, справочники, офлайн-кэш и синхронизация, роли, отчёты, тесты (unit, Espresso). 
1. Анализ технического задания

Целью разработки является создание мобильного приложения, обеспечивающего пользователям возможность просмотра и управления данными об успеваемости (группы, дисциплины, студенты, оценки и посещаемость), выставления оценок через журнал, формирования отчётов (Excel/PDF) с отправкой на e-mail или сохранением на устройство, а также работу с данными в офлайн-режиме с последующей синхронизацией.
Описание системы: Система представляет собой распределённое приложение из мобильного приложения на Kotlin (View и XML-разметки, ViewBinding), серверной части (Spring Boot, Java 17) и базы данных PostgreSQL. Приложение кэширует справочники (факультеты, группы, дисциплины, студенты) в Room и синхронизирует их с сервером при входе и по расписанию (WorkManager); экраны админ-панели (списки пользователей, студентов, групп и т.д.) загружают данные с API при открытии и при возврате с экрана редактирования.

•	Деканат: главный экран — вкладки «Группы», «Дисциплины», «Оценки», «Профиль»; админ-панель без раздела «Пользователи» и «Факультеты»: Студенты, Группы, Кафедры, Дисциплины, Преподаватели, Журнал, Отчёты.
•	Администратор: главный экран — «Ещё» (О системе, Админ-панель) и «Профиль»; в админ-панели все разделы: Пользователи, Студенты, Группы, Факультеты, Кафедры, Дисциплины, Преподаватели, Журнал, Отчёты.
Функциональные требования (реализованное):
1. Модуль аутентификации: вход по логину и паролю с проверкой на сервере (JWT); восстановление пароля по e-mail; хранение токена на устройстве в SharedPreferences (SessionManager).
2. Модуль справочников и журнала: CRUD по пользователям, студентам, группам, факультетам, кафедрам, дисциплинам, преподавателям через экраны админ-панели (данные с API, списки обновляются при возврате в onResume). 2. Разработка архитектуры
Архитектура является фундаментом разработки, обеспечивая масштабируемость, сопровождаемость, тестируемость и надёжность приложения. В основе решения — комбинация Clean Architecture и MVVM (Model–View–ViewModel), что даёт чёткое разделение ответственности между компонентами и помогает соблюдать принципы SOLID.
2.1. Цели и ограничения архитектуры
Цели:
•	Масштабируемость: возможность безопасного добавления нового функционала и модулей.
•	Тестируемость: изоляция бизнес-логики (use cases) для unit-тестирования.
•	Поддерживаемость: очевидное разделение слоёв для упрощения изменений.
•	Надёжность: корректная работа при нестабильном соединении (офлайн-режим за счёт локального кэша, повторная синхронизация).
•	Безопасность: хранение токена в SharedPreferences (SessionManager); сетевые запросы по HTTPS с JWT в заголовке Authorization.
Ограничения:
•	Целевая платформа: Android 8.0 (API 26) и выше.
•	Язык разработки: Kotlin.
•	UI: View и XML-разметки, ViewBinding (без Jetpack Compose).
Выбранная многослойная архитектура состоит из трёх основных уровней:
1. Presentation Layer (слой представления)
•	Компоненты: Activity, Fragment, ViewBinding (XML), ViewModel (AndroidX ViewModel), состояние UI (StateFlow / LiveData).
•	Отвечает за отображение, обработку пользовательских событий и привязку UI к состоянию.
•	Взаимодействует с Domain-слоем через Use Cases; не знает деталей реализации репозиториев и источников данных.
2. Domain Layer (слой бизнес-логики)
•	Компоненты: Use Cases (интеракторы), интерфейсы репозиториев, доменные модели.
•	Не зависит от платформенных библиотек — удобен для unit-тестирования.
•	Содержит правила синхронизации, валидации и сценарии работы с данными (например, при офлайне — приоритет кэша).

3. Data Layer (слой данных)
•	Компоненты: реализации репозиториев, источники данных (сеть, локальная БД), мапперы DTO ↔ сущности.
•	Remote: Retrofit + OkHttp для REST API, JWT в заголовке Authorization (интерцептор).
•	Local: Room для кэша справочников (факультеты, группы, дисциплины, студенты) и при необходимости оценок; SessionManager на базе SharedPreferences для токена и данных сессии.
•	Фоновая синхронизация: WorkManager (SyncWorker) раз в 15 минут вызывает SyncCatalogUseCase → SyncManager загружает с API факультеты, группы, дисциплины, студентов и сохраняет в Room.
Направление зависимостей строго регламентировано: Presentation → Domain ← Data. Domain не зависит от Presentation и Data, что обеспечивает его изоляцию и удобство тестирования.
Основные паттерны и практики:
•	Репозитории (Repository pattern): интерфейсы в Domain, реализации в Data (сеть + Room).
•	Use Cases: атомарные единицы бизнес-логики, вызываются из ViewModel и из фоновых воркеров.
•	DTO и мапперы: преобразование между сетевыми моделями, сущностями Room и доменными моделями.
•	Корутины и Flow: асинхронность и потоковые состояния (StateFlow) в ViewModel.
•	Синхронизация: клиент загружает данные с сервера (GET-запросы), обновляет локальную БД (Room); при отсутствии сети отображаются кэшированные данные.
•	Безопасность: JWT хранится в SharedPreferences (SessionManager); при запросах токен добавляется в заголовок Authorization (интерцептор OkHttp).
Офлайн и стратегия синхронизации:
•	Локальные справочники и оценки кэшируются в Room.
•	WorkManager (CoroutineWorker) выполняет фоновую синхронизацию справочников по расписанию; при ошибке — повторные попытки (Result.retry()).
Тестируемость:
•	ViewModel тестируются unit-тестами (JUnit, Mockito); инструментальные тесты — экран входа и навигация (Espresso). Интеграционные тесты Data-слоя с MockWebServer в проекте не реализованы.
Дополнительные замечания:
•	Зависимости внедряются через конструкторы (при необходимости можно добавить Hilt или другой DI).
Архитектура спроектирована так, чтобы её можно было расширить: очередь отложенных операций, дополнительные отчёты или уведомления в будущем.

2.2. Детализация слоёв
1. Presentation Layer (слой представления)
Компоненты:
•	UI (Activity, Fragment, ViewBinding): экраны на XML-разметках, отрисовка UI и передача пользовательских событий во ViewModel.
•	ViewModel: androidx.lifecycle.ViewModel. Содержит наблюдаемое состояние экрана (StateFlow / LiveData), обрабатывает действия пользователя и вызывает Use Cases из Domain-слоя. ViewModel переживает смену конфигурации.
•	State / UiState: неизменяемые data-классы или типы, описывающие состояние UI.

Поток данных в UI:
•	ViewModel предоставляет StateFlow или LiveData.
•	View (Fragment/Activity) подписывается на этот поток (collect / observe).
•	При изменении состояния UI обновляется.
•	Действие пользователя вызывает метод во ViewModel, который обновляет состояние и/или запускает Use Case.

2. Domain Layer (слой бизнес-логики)
Этот слой — чистый Kotlin без зависимостей от Android SDK.
Компоненты:
•	Use Cases (интеракторы): классы, инкапсулирующие сценарии (синхронизация справочников, загрузка оценок студента и т.д.). Каждый Use Case отвечает за одну задачу и использует интерфейсы репозиториев.
•	Интерфейсы репозиториев: контракты доступа к данным (ISyncCatalog и др.).
•	Доменные модели: бизнес-сущности (Faculty, Group, Subject, Student, Grade и т.п.), независимые от способа хранения или передачи.
Назначение: содержать бизнес-логику — правила синхронизации, валидацию, приоритет кэша при офлайне.
3. Data Layer (слой данных)
Компоненты:
•	Реализации репозиториев: реализуют интерфейсы Domain, решают, откуда брать данные (локально или с сети), содержат стратегию кэширования и синхронизации.
•	Источники данных:
•	Удалённый доступ: взаимодействие с REST API через Retrofit + OkHttp; преобразование DTO в доменные модели или сущности Room; JWT в заголовках через OkHttp-interceptor (SessionManager, AuthInterceptor).
•	Локальный доступ: Room (сущности и DAO для факультетов, групп, дисциплин, студентов, пользователей, преподавателей, оценок, типов оценок); SessionManager (SharedPreferences) для токена и настроек сессии.
•	Модели данных: DTO для API, Entity для Room.
•	Мапперы: преобразование между DTO / Entity и доменными моделями.
Особенности синхронизации и офлайна:
•	Справочники и оценки хранятся в Room; при наличии сети данные загружаются с сервера и обновляют локальную БД.
•	SyncManager загружает только факультеты, группы, дисциплины и студентов; преподаватели, типы оценок и оценки в локальной БД заполняются тестовыми данными при первом создании БД (DatabaseCallback) или приходят с API при открытии соответствующих экранов.
Безопасность:
•	JWT хранится в SharedPreferences (SessionManager); интерцептор OkHttp добавляет заголовок Authorization.
•	Сетевые вызовы по HTTPS.
Тестируемость:
•	SyncCatalogUseCase вызывается из SyncWorker; отдельные unit-тесты Use Case в проекте не выделены.
•	ViewModel покрыты unit-тестами (AuthViewModelTest, GradeViewModelTest, UserViewModelTest, AdminViewModelTest) с моками репозиториев; UI-тесты — AuthScreenTest и MainNavigationTest (Espresso).
Паттерны и лучшие практики:
•	Репозитории: интерфейсы в Domain, реализации в Data.
•	Use Cases как единицы бизнес-логики, вызываемые ViewModel и воркерами.
•	DTO ↔ Entity ↔ Domain мапперы.
•	Внедрение зависимостей через конструкторы.
•	Корутины и Flow для асинхронности и потоков состояний.
•	Экономия трафика: кэш в Room, выборочная синхронизация по расписанию.

Соответствие серверу:
•	DTO на клиенте соответствуют серверным контрактам (ответы API: списки факультетов, групп, дисциплин, студентов, оценок и т.д.).
•	Эндпоинты: /api/auth, /api/faculties, /api/groups, /api/subjects, /api/students, /api/grades и др. — используются при проектировании API-клиента и маппинге.
•	2.3. Ключевые потоки данных

1. Просмотр групп и дисциплин на главном экране (из Room)
•	GroupsFragment и SubjectsFragment получают данные из AdminViewModel → StudentRepository → Room (groupDao, subjectDao). Данные в Room обновляются при синхронизации (при входе в MainActivity вызывается SyncManager.sync(), а также по кнопке «Обновить» в меню и по расписанию WorkManager).
2. Списки в админ-панели (прямые запросы к API)
•	UsersListActivity, StudentsListActivity, GroupsListActivity, FacultiesListActivity, DepartmentsListActivity, SubjectsListActivity, TeachersListActivity при создании и в onResume() вызывают загрузку с API (Retrofit) и отображают результат в RecyclerView. Локальный кэш Room для этих экранов не используется; после возврата с экрана добавления/редактирования список перезагружается в onResume().
3. Синхронизация справочников (при старте и по расписанию)
•	При входе в MainActivity в корутине вызывается SyncManager.sync(): запросы GET /api/faculties, /api/groups, /api/subjects, /api/users/students и сохранение ответов в Room (facultyDao, groupDao, subjectDao, studentDao).
•	WorkManager запускает SyncWorker раз в 15 минут (PeriodicWorkRequestBuilder); SyncWorker вызывает SyncCatalogUseCase → SyncManager.sync(). При сбое возвращается Result.retry().
4. Журнал и оценки
•	JournalListActivity загружает список занятий через GET /api/journal/lesson-records с фильтрами (дата, предмет, группа, тип). По клику открывается JournalActivity с параметрами занятия; список студентов группы и текущие оценки загружаются с API; сохранение оценок и посещаемости — через API. Списки на экране после возврата обновляются в onResume().

Диаграммы Use Case и физическую модель данных можно привести в приложениях к пояснительной записке. 
3. Разработка и реализация системы
3.1. Создание баз данных
Серверная БД — «источник правды»: хранит полные данные о пользователях, факультетах, группах, дисциплинах, студентах, преподавателях, типах оценок, назначенных курсах, оценках, посещаемости, итоговых оценках и служебных таблицах (отчёты, токены). Обеспечивает транзакционную целостность, индексы для быстрого поиска и миграции через Flyway.
Локальная (мобильная) БД — облегчённый кэш на устройстве для офлайн-режима: просмотр ранее загруженных справочников и оценок без сети. Не копирует всю серверную модель, хранит только необходимые для работы приложения сущности и ограничивает объём чувствительных данных.
Технологии и инфраструктура:
• PostgreSQL; миграции Flyway (V1__init_schema_and_seed.sql — первоначальная схема и начальные данные, далее V2–V10 — представления, триггеры, посещаемость, итоги, отчёты, токены и т.д.).
• JPA/Hibernate (Spring Data JPA), сущности (User, Faculty, Department, StudyGroup, Teacher, Student, Subject, GradeType, AssignedCourse, Grade, FinalGrade, Attendance и др.).
• Конфигурация в application.yml: datasource (URL, учётные данные), jwt.secret, spring.mail и пр.
Основные таблицы:
1. users — пользователь
• Назначение: учётные записи (логин, пароль, роль).
• Ключевые поля: user_id, login, password_hash, email, last_name, first_name, middle_name, role (ADMIN, TEACHER, STUDENT, DEANERY), is_active, created_at, updated_at.
• Связи: студент и преподаватель связаны с users через user_id.
2. faculties — факультет
• Назначение: справочник факультетов.
• Ключевые поля: faculty_id, name, dean_name, created_at.
• Связи: группы и кафедры ссылаются на faculty_id.
3. departments — кафедра
• Назначение: справочник кафедр.
• Ключевые поля: department_id, name, faculty_id, head_name, created_at.
• Связи: @ManyToOne faculty; преподаватели ссылаются на department_id.
4. study_groups — учебная группа
• Назначение: справочник групп.
• Ключевые поля: group_id, name, faculty_id, admission_year, specialization, created_at.
• Связи: @ManyToOne faculty; студенты и назначенные курсы ссылаются на group_id.
5. teachers — преподаватель
• Назначение: привязка пользователя к кафедре и должности.
• Ключевые поля: teacher_id, user_id, department_id, academic_degree, position, created_at.
• Связи: @ManyToOne user, department; assigned_courses ссылаются на teacher_id.
6. students — студент
• Назначение: привязка пользователя к группе и зачётной книжке.
• Ключевые поля: student_id, user_id, group_id, record_book_number, admission_year, birth_date, phone_number, address, created_at.
• Связи: @ManyToOne user, study_group; оценки и посещаемость ссылаются на student_id.
7. subjects — дисциплина
• Назначение: справочник дисциплин.
• Ключевые поля: subject_id, name, code, credits, total_hours, lecture_hours, practice_hours, lab_hours, control_type (EXAM, CREDIT, DIFF_CREDIT), description, created_at.
• Связи: оценки, посещаемость, назначенные курсы ссылаются на subject_id.
8. grade_types — тип оценки
• Назначение: виды работ (лекция, практика, лабораторная, экзамен и т.д.) и их веса.
• Ключевые поля: type_id, name, weight, description.
• Связи: оценки ссылаются на grade_type_id
9. assigned_courses — назначенный курс
• Назначение: связь преподаватель — группа — дисциплина на учебный год/семестр.
• Ключевые поля: assignment_id, teacher_id, group_id, subject_id, academic_year, semester, created_at.
• Ограничение: уникальность (teacher_id, group_id, subject_id, academic_year, semester).
10. grades — оценка
• Назначение: текущие оценки по студенту, дисциплине, типу работы.
• Ключевые поля: grade_id, student_id, subject_id, assignment_id, grade_type_id, grade_value, grade_date, semester, academic_year, comment, work_type, created_at, updated_at.
• Связи: @ManyToOne student, subject, assigned_course, grade_type.
11. attendance — посещаемость
• Назначение: отметки присутствия/отсутствия по дате занятия.
• Ключевые поля: attendance_id, student_id, subject_id, assignment_id, lesson_date, present, semester, academic_year, comment, created_at.
• Ограничение: уникальность (student_id, subject_id, lesson_date).
12. final_grades — итоговая оценка
• Назначение: экзамен/зачёт за семестр по дисциплине.
• Ключевые поля: final_grade_id, student_id, subject_id, semester, academic_year, final_grade, is_credited, is_academic_debt, approved_by, approved_at, created_at.

 Локальная (мобильная) база данных (Room)
Назначение — кэш для главного экрана: экраны «Группы» и «Дисциплины» читают данные из Room; данные обновляются при синхронизации (при входе в MainActivity, по кнопке «Обновить» и по расписанию WorkManager раз в 15 минут). При первом создании БД выполняется DatabaseCallback.fillTestData (тестовые факультеты, группы, дисциплины, пользователи, студенты, типы оценок, оценки). SyncManager записывает в Room только факультеты, группы, дисциплины и студентов.
Таблицы:
1. users (UserEntity) — пользователи (локальная копия для отображения)
• Назначение: кэш пользователей для списков и профиля.
• Ключевые поля: userId (PK), login, email, lastName, firstName, middleName, role, isActive, createdAt.
• Используется при отображении списков и профиля.
2. faculties (FacultyEntity) — факультеты
• Назначение: кэш справочника факультетов.
• Ключевые поля: facultyId (PK), name, deanName, createdAt.
3. study_groups (GroupEntity) — учебные группы
• Назначение: кэш справочника групп.
• Ключевые поля: groupId (PK), name, facultyId, admissionYear, specialization, createdAt.
• Связи: внешний ключ на faculties.
4. subjects (SubjectEntity) — дисциплины
• Назначение: кэш справочника дисциплин.
• Ключевые поля: subjectId (PK), name, code, credits, totalHours, lectureHours, practiceHours, labHours, controlType, description, createdAt.
5. students (StudentEntity) — студенты
• Назначение: кэш списка студентов с привязкой к группе и пользователю.
• Ключевые поля: studentId (PK), userId, groupId, recordBookNumber, admissionYear, birthDate, phoneNumber, address, createdAt.
• Связи: внешние ключи на users, study_groups.

6. teachers (TeacherEntity) — преподаватели
• Назначение: кэш списка преподавателей.
• Ключевые поля: teacherId (PK), userId, departmentId, academicDegree, position, createdAt.
• Связи: внешние ключи на users, departments (при наличии локальной таблицы кафедр или только userId).

7. grade_types (GradeTypeEntity) — типы оценок
• Назначение: кэш типов работ (лекция, практика, экзамен и т.д.).
• Ключевые поля: typeId (PK), name, weight, description.

8. grades (GradeEntity) — оценки
• Назначение: кэш оценок для отображения в журнале и у студента.
• Ключевые поля: gradeId (PK, autoGenerate для локальных), studentId, subjectId, gradeTypeId, gradeValue, gradeDate, semester, academicYear, comment, workType, createdAt, updatedAt.
• Связи: внешние ключи на students, subjects.

База Room (StudentDatabase) создаётся с версией 1, при первом запуске при необходимости заполняется тестовыми данными через DatabaseCallback (для разработки/демо); в рабочем режиме данные поступают с сервера через SyncManager и сохраняются в соответствующие DAO.

3.2. Программирование системы
Мобильное приложение реализовано на платформе Android на языке Kotlin с архитектурной комбинацией Clean Architecture и MVVM, что обеспечивает модульность, тестируемость и сопровождаемость кода; приложение рассчитано на работу в условиях нестабильного сетевого соединения и поддерживает офлайн-режим за счёт локального кэша и фоновой синхронизации при восстановлении связи. Архитектура организована по слоям Presentation → Domain ← Data: сущности оформлены как Kotlin data-классы с мапперами DTO ↔ Entity ↔ Domain; слой представления построен на Activity, Fragment, ViewBinding (XML-разметки) и ViewModel с StateFlow/LiveData; Domain содержит use cases и интерфейсы репозиториев (например, ISyncCatalog); Data реализует репозитории, доступ к API (Retrofit + OkHttp с интерцептором для добавления JWT) и локальное хранилище (Room) с стратегией загрузки данных с сервера и обновления кэша.

Токен и данные сессии хранятся в SharedPreferences (SessionManager). Синхронизация справочников (факультеты, группы, дисциплины, студенты) выполняется при входе в MainActivity и по расписанию WorkManager (15 минут); SyncManager запрашивает API и сохраняет данные в Room. Экраны админ-панели (списки пользователей, студентов, групп и т.д.) загружают данные с API при открытии и в onResume(); журнал (JournalListActivity, JournalActivity) работает с API (список занятий с фильтрами, сохранение оценок и посещаемости). Отчёты формируются на сервере; клиент скачивает файл или запрашивает отправку на e-mail текущего пользователя.
UI: экран входа (AuthActivity), главный экран (MainActivity) с нижней навигацией в зависимости от роли — у студента «Оценки» и «Профиль»; у деканата «Группы», «Дисциплины», «Оценки», «Профиль»; у преподавателя и администратора пункт «Ещё» (у преподавателя — Журнал, Отчёты; у администратора — О системе, Админ-панель) и «Профиль». Админ-панель (AdminPanelActivity): разделы по ролям (администратор — все разделы включая Пользователи и Факультеты; деканат — без них; преподаватель — Студенты, Журнал, Отчёты). Формы добавления/редактирования для всех справочников; при возврате список обновляется в onResume(). Разграничение по ролям реализовано в UI и на сервере (@PreAuthorize).
Валидация и бизнес-правила выполняются на сервере. Сборка — Gradle (Kotlin DSL); тесты — unit ViewModel (JUnit, Mockito), инструментальные AuthScreenTest и MainNavigationTest (Espresso); Dokka подключена для документации.
В результате получено мобильное приложение, интегрированное с Spring Boot и PostgreSQL (Flyway), с кэшем справочников в Room, фоновой синхронизацией (WorkManager), журналом занятий с фильтрами, CRUD справочников и отчётами Excel/PDF (скачивание или отправка на e-mail).

Полный листинг исходного кода мобильного приложения представлен в приложении В, скриншоты пользовательского интерфейса — в приложении Б.



 
4. Тестирование системы
Тестирование мобильного приложения для контроля успеваемости студентов направлено на проверку корректности бизнес-логики во ViewModel, надёжности отображения данных и стабильности пользовательского опыта при изменениях. Основные цели — проверить обработку входных и выходных данных во ViewModel и репозиториях, верифицировать переходы между состояниями UI (Loading, Success, Error), проверить поведение при граничных условиях и ошибках (пустые поля входа, ошибка авторизации, различие сценариев по ролям), а также зафиксировать ожидаемое поведение в виде автотестов.
В проекте принят модульный подход к тестированию клиентской логики в изоляции с использованием JUnit (JVM-тесты) и AndroidX Test (инструментальные тесты). Для мокирования зависимостей (репозиториев) применяется Mockito с аннотациями и расширениями (MockitoAnnotations.openMocks, @Mock), что позволяет не задействовать реальную сеть и базу данных в unit-тестах и добиваться детерминированности. Для асинхронной логики и корутин используется kotlinx-coroutines-test: runTest, StandardTestDispatcher, а также Dispatchers.setMain / resetMain для привязки корутин к тестовому диспетчеру; для LiveData в тестах применяется androidx.arch.core:core-testing (InstantTaskExecutorRule), чтобы выполнение задач было синхронным. Проверки выполняются средствами JUnit (assertEquals, assertTrue, assertNull). Вспомогательный класс LiveDataTestUtil (getOrAwaitValue) используется при необходимости дождаться значения LiveData в тестах.
При написании тестов соблюдаются принципы изоляции, детерминированности и читаемости. Каждый unit-тест проверяет одну конкретную сценарию; входные данные задаются через моки и тестовые сущности (GradeEntity, StudentEntity, UserEntity и т.д.). Покрываются позитивные сценарии (успешная загрузка, успешный вход) и обработка ошибок (пустые поля, неверный пароль, различие поведения по роли пользователя). Тесты строятся по схеме Arrange-Act-Assert и имеют понятные имена (loadAllGrades_withoutStudentFilter_loadsAllGradesAndAverage, login with empty fields emits Error и т.п.).
Структура тестов зеркалирует структуру приложения: для ViewModel выделены отдельные тестовые классы — AuthViewModelTest, GradeViewModelTest, UserViewModelTest, AdminViewModelTest. В @Before выполняется инициализация моков (MockitoAnnotations.openMocks), установка тестового диспетчера (Dispatchers.setMain(testDispatcher)) и настройка поведения репозитория (whenever, coEvery для возврата flowOf(...) или фиксированных данных). В @After вызывается Dispatchers.resetMain(). В AuthViewModelTest используется реализация-заглушка IAuthRepository (FakeAuthRepository), возвращающая заданный Result, что позволяет проверить переходы состояний входа (Error при пустых полях, Success при успехе, Error при ошибке сервера). В GradeViewModelTest, UserViewModelTest и AdminViewModelTest мокируется StudentRepository: проверяется загрузка оценок без фильтра и с фильтром по роли STUDENT, загрузка оценок по студенту, загрузка списка пользователей, выбор пользователя по id, сброс состояния при logout, загрузка статистики для админ-панели (количество пользователей, студентов, групп, факультетов, дисциплин).
Практические проверки включают: для AuthViewModel — валидацию пустых полей, успешный вход (эмит Success с данными пользователя), вход при ошибке (эмит Error с сообщением); для GradeViewModel — загрузку всех оценок при роли администратора, фильтрацию по studentId при роли студента, загрузку оценок выбранного студента и среднего балла; для UserViewModel — загрузку списка пользователей, обновление currentUser при выборе по id, сброс состояния при logout; для AdminViewModel — загрузку счётчиков для отображения на админ-панели. Обязательно проверяются корректные типы UI-состояний (Success, Error, Initial) и ожидаемые значения в state/value после advanceUntilIdle().
Инструментальные тесты дополняют модульные проверки на уровне UI и навигации. Для них используются AndroidJUnit4, ActivityScenario и Espresso. AuthScreenTest проверяет отображение экрана входа: наличие полей логина и пароля и кнопки входа (onView(withId(...)).check(matches(isDisplayed()))). MainNavigationTest проверяет главный экран и навигацию по вкладкам: перед каждым тестом устанавливается тестовая сессия (SessionTestHelper.setTestSession), чтобы приложение открыло MainActivity без реального входа; проверяется отображение нижней навигации и контента, списка групп; переход на вкладку «Дисциплины» и отображение списка дисциплин; переход на вкладку «Оценки» и отображение списка оценок; переход в профиль и отображение кнопки выхода; сценарий выхода — нажатие кнопки выхода и возврат на экран входа (отображение поля логина). После тестов сессия очищается (clearTestSession в @After). Таким образом проверяется корректность отображения основных экранов и навигации между ними в условиях эмулятора или устройства.
Тесты служат как средством контроля качества, так и документацией ожидаемого поведения. Покрытие сфокусировано на ключевых сценариях: аутентификация (вход, валидация, ошибки), загрузка и отображение данных по ролям (оценки, пользователи, статистика админки), навигация и выход. Описанный набор unit- и инструментальных тестов даёт уверенность в корректности клиентской логики и упрощает поддержку и доработку приложения.
 
5. Руководство пользователя
 5.1. Регистрация в системе
В мобильном приложении экран регистрации не реализован. Новых пользователей создаёт администратор в админ-панели (раздел «Пользователи» → кнопка «Добавить» или выбор пользователя для редактирования). Пользователю выдаются логин и пароль; вход выполняется через экран входа (п. 5.2).
 5.2. Вход в систему
1. Запустите приложение. Откроется экран входа (логин, пароль, кнопки «Войти» и «Забыли пароль»).
2. Введите логин и пароль (выданные администратором).
3. Нажмите «Войти». При пустых полях или неверных данных отобразится сообщение об ошибке.
После входа откроется главный экран. Навигация по ролям: студент — вкладки «Оценки», «Профиль»; деканат — «Группы», «Дисциплины», «Оценки», «Профиль»; преподаватель и администратор — пункт «Ещё» и «Профиль» (у преподавателя в «Ещё» — Журнал и Отчёты; у администратора — О системе и Админ-панель). Из профиля доступны кнопки «Админ-панель» (для ADMIN, DEANERY, TEACHER), «Сменить пароль» и «Выйти».
5.3. Работа с данными приложения
 Просмотр групп, дисциплин и оценок

Вкладки «Группы» и «Дисциплины» (доступны роли деканат) отображают данные из локального кэша (Room); обновление — по кнопке «Обновить» в меню главного экрана или автоматически при входе и по расписанию. Вкладка «Оценки» показывает для студента его оценки и средний балл; данные загружаются с API. Списки в админ-панели (Пользователи, Студенты, Группы и т.д.) загружаются с сервера при открытии экрана и при возврате с экрана редактирования.

 Профиль пользователя
1. Перейдите на вкладку «Профиль».
2. На экране отображаются данные текущего пользователя (логин, e-mail, роль и т.д.).
3. Доступные действия (в зависимости от роли и реализации):
   • В профиле (ProfileActivity): смена пароля (по e-mail), кнопка «Админ-панель» (для ADMIN, DEANERY, TEACHER), выход из системы. Журнал и отчёты открываются из админ-панели или для преподавателя из пункта «Ещё» на главном экране.
4. Нажмите «Выйти», чтобы завершить сессию — будет открыт экран входа.

 Админ-панель (для ролей администратор, деканат, преподаватель)
Админ-панель открывается из профиля (кнопка «Админ-панель») для ролей ADMIN, DEANERY, TEACHER. Разделы по ролям: администратор — Пользователи, Студенты, Группы, Факультеты, Кафедры, Дисциплины, Преподаватели, Журнал, Отчёты; деканат — те же, кроме Пользователи и Факультеты; преподаватель — Студенты, Журнал, Отчёты. Списки загружаются с сервера при открытии и при возврате с формы; по кнопке «Добавить» или по клику на элемент открывается форма добавления/редактирования.
Журнал оценок и посещаемости
1. Выберите «Журнал» в админ-панели (или «Журнал» в пункте «Ещё» для преподавателя).
2. На экране списка занятий задайте фильтры: дата с/по, предмет, группа, тип занятия. Нажмите применить — отобразится список занятий.
3. Нажмите на занятие — откроется экран занятия (дата, группа, предмет, тип). Список студентов группы; для каждого — оценка и отметка посещаемости. Сохранение через кнопку на экране; данные отправляются на сервер. При необходимости используется экран добавления оценки (AddGradeActivity).

Формирование отчётов
1. В админ-панели выберите «Отчёты».
2. Выберите тип отчёта: по группе, по предмету или по студенту; укажите группу, предмет или студента.
3. Выберите формат (Excel или PDF) и действие: скачать файл на устройство или отправить на e-mail текущего пользователя (сервер отправляет на сохранённый в учётной записи адрес). Результат отображается сообщением.

 Выход из системы
1. Перейдите на вкладку «Профиль».
2. Нажмите кнопку «Выйти» (или «Выход»).
3. Сессия будет завершена, токен удалён; откроется экран входа.

 
6. Руководство программиста
Мобильное приложение разработано по принципам Clean Architecture с применением паттерна MVVM (Model–View–ViewModel). Система разделена на три основных слоя: Presentation Layer (UI и ViewModel), Domain Layer (бизнес-логика и интерфейсы репозиториев) и Data Layer (источники данных — сеть и локальное хранилище). Такая многослойная организация обеспечивает чёткое разделение ответственности, упрощает тестирование и сопровождение кода, а также даёт возможность плавно масштабировать функционал.
Проект организован по функциональным областям с пакетной структурой внутри одного модуля app: data (реализации репозиториев, remote — API и DTO, local — Room: сущности, DAO, база данных, sync — SyncManager), domain (use cases, интерфейсы вроде ISyncCatalog), viewmodel (AuthViewModel, GradeViewModel, UserViewModel, AdminViewModel и др.), ui (Activity и Fragment по экранам: AuthActivity, MainActivity, группы, дисциплины, оценки, профиль, admin — списки и формы справочников, журнал, отчёты), worker (WorkManager — SyncWorker), service (Firebase и др.), а также util при необходимости. Навигация по коду и рефакторинг упрощаются за счёт явного разделения на data/domain и presentation.
Язык разработки — Kotlin; минимальная поддерживаемая версия Android — API 26 (Android 8.0). Сборка выполняется через Gradle с использованием Kotlin DSL; модуль app содержит весь клиентский код. Внедрение зависимостей выполняется через конструкторы (ViewModelProvider с фабриками, передача репозиториев и API в ViewModel и сервисы); при необходимости может быть добавлен Hilt или другой DI-фреймворк.
UI реализован на View и XML-разметках с ViewBinding (архитектура на Activity и Fragment). Главный экран — MainActivity с нижней навигацией и контейнером для фрагментов (группы, дисциплины, оценки, профиль). Presentation-слой содержит ViewModel (AndroidX ViewModel), управляющие состоянием через StateFlow / LiveData; Activity и Fragment подписываются на эти потоки (collect / observe) и обновляют интерфейс при изменении данных. Навигация между экранами выполняется через Intent (запуск Activity: админ-панель, журнал, отчёты, формы добавления/редактирования). Интерфейс следует рекомендациям Material Design и может быть адаптирован под разные размеры экранов.
Data Layer построен по паттерну Repository: интерфейсы репозиториев и контракты (например, ISyncCatalog) определены в domain, реализации — в data. Локальное хранение осуществляется через Room (сущности FacultyEntity, GroupEntity, SubjectEntity, StudentEntity, UserEntity, GradeEntity, GradeTypeEntity и соответствующие DAO); настройки сессии и хранение токена — через SharedPreferences (SessionManager). Сетевой слой основан на Retrofit и OkHttp с интерцептором (AuthInterceptor) для добавления заголовка Authorization (JWT). Данные с сервера маппятся из DTO в сущности Room и при необходимости в доменные модели.
Domain Layer содержит интерфейс ISyncCatalog и use case SyncCatalogUseCase, инкапсулирующий сценарий синхронизации справочников (факультеты, группы, дисциплины, студенты). Use case вызывается из SyncWorker и при ручном обновлении; все операции оформлены как suspend-функции и работают с корутинами. Валидация (например, проверка полей входа) может выполняться во ViewModel или в domain; Presentation-слой остаётся тонким и тестируемым.
Офлайн-режим и синхронизация организованы через локальный кэш в Room: при наличии сети данные загружаются с сервера (SyncManager запрашивает API и сохраняет результаты в Room). При старте приложения и по расписанию запускается WorkManager (SyncWorker), который вызывает SyncCatalogUseCase и тем самым обновляет справочники; при ошибке сети возвращается Result.retry() для повторной попытки. Стратегия отображения: данные читаются из Room (кэш), при открытии экранов при необходимости выполняется запрос к API и обновление локальной БД (например, загрузка оценок студента). Очереди отложенных операций (pending_sync) и черновиков в проекте не реализованы — синхронизация идёт в направлении сервер → клиент.
Обработка ошибок и состояния UI унифицированы через sealed-классы состояний (AuthUiState, GradeUiState, UserUiState и т.д.): Idle, Loading, Success, Error. Это позволяет единообразно обрабатывать сетевые ошибки, ошибки аутентификации (401/403) и отображать пользователю понятные сообщения. Для реактивных потоков используется Flow и StateFlow; для тестирования корутин — kotlinx-coroutines-test (runTest, StandardTestDispatcher).
Управление сборкой и зависимостями выполняется через Gradle (Kotlin DSL); в build.gradle.kts подключены зависимости для Room, Retrofit, OkHttp, WorkManager, тестирования (JUnit, Mockito, Espresso) и Dokka для документации. При необходимости может быть настроен CI для прогона тестов и статического анализа.
В результате архитектура даёт удобную основу для расширения: добавление очереди отложенных операций, дополнительных отчётов, push-уведомлений или доработка синхронизации — всё это можно интегрировать в существующие слои благодаря чёткому разделению и соблюдению контрактов между Presentation, Domain и Data.
 
Заключение
В ходе выполнения проекта разработано и реализовано мобильное приложение «Контроль успеваемости» (система учёта успеваемости студентов) на языке Kotlin для платформы Android в связке с серверной частью на Spring Boot. Приложение предназначено для централизованного учёта академических данных: справочники (факультеты, кафедры, группы, дисциплины, студенты, преподаватели), электронный журнал оценок и посещаемости, формирование отчётов (Excel/PDF) с отправкой на e-mail или сохранением на устройство, а также эффективная работа при нестабильной сети за счёт локального кэша и фоновой синхронизации.
Разработанное решение полностью соответствует требованиям технического задания: обеспечена безопасная аутентификация и авторизация (JWT, Spring Security), надёжное хранение паролей на сервере (BCrypt), разграничение ролей (администратор, деканат, преподаватель, студент) и защита эндпоинтов. На клиенте токен хранится в SharedPreferences (SessionManager); для офлайн-режима реализовано локальное хранилище (Room) и фоновая синхронизация справочников через WorkManager (SyncWorker, SyncCatalogUseCase).
Ключевым достижением проекта стала интеграция мобильного клиента с серверной частью на Spring Boot и PostgreSQL (миграции Flyway): согласованные DTO и REST API, CRUD по факультетам, группам, дисциплинам, студентам, преподавателям, пользователям, ведение журнала занятий (дата, тип занятия, группа, дисциплина, оценки и посещаемость), формирование отчётов и уведомление пользователя по e-mail при входе. Приложение прошло тестирование основных модулей (unit-тесты ViewModel, инструментальные тесты экрана входа и навигации с Espresso), подтвердив соответствие функциональным требованиям.
Архитектурные решения — Clean Architecture и MVVM с использованием Android Jetpack (ViewModel, StateFlow/LiveData), View и XML-разметок с ViewBinding, Coroutines, Retrofit/OkHttp и модульной структуры (data, domain, ui, viewmodel, worker) — обеспечивают сопровождаемость кода и возможность расширения (дополнительные отчёты, push-уведомления через FCM, очередь отложенных операций). Проект демонстрирует практическую применимость клиент-серверной АИС для учёта успеваемости в образовательном учреждении с централизованным хранением данных, удобным журналом и отчётностью.
