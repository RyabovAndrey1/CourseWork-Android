# 5. Тестирование системы

Тестирование сервисного слоя серверной части веб-приложения «Учёт успеваемости студентов» направлено на проверку корректности бизнес-логики формирования отчётов, надёжности приложения и удобства последующего рефакторинга. Основные цели тестирования — проверить формирование отчётов по группе, дисциплине и студенту при различных входных данных, поведение при отсутствии сущностей (например, студент не найден), вызовы репозиториев и возврат ресурсов (Excel/PDF), а также зафиксировать ожидаемое поведение сервисов в виде набора автоматических тестов.

В проекте принят модульный подход к тестированию сервисов в изоляции с использованием JUnit 5 в качестве фреймворка для написания и организации тестов. Для мокирования зависимостей (репозиториев: GradeRepository, StudentRepository, GroupRepository, AttendanceRepository, SubjectRepository) применяется Mockito с аннотациями `@ExtendWith(MockitoExtension.class)`, `@Mock` и `@InjectMocks`, что позволяет не обращаться к реальной БД в юнит-тестах и получать детерминированный результат. Для утверждений используется AssertJ (`assertThat(...)`), что улучшает читаемость тестов. Для проверки загрузки контекста Spring и профиля приложения используются тесты с аннотацией `@SpringBootTest` и профилем `test` (in-memory H2, отключённый Flyway, отключённая реальная отправка почты).

При написании тестов соблюдаются принципы изоляции, детерминированности и читаемости. Каждый юнит-тест проверяет один сценарий работы сервиса; входные данные задаются через моки (Optional.of(entity) или Optional.empty(), пустые списки); покрываются позитивные сценарии (возврат непустого ресурса при существующих группе, студенте, предмете) и обработка ошибок (выброс RuntimeException при отсутствии студента). Тесты организованы по паттерну Arrange–Act–Assert; имена методов отражают ожидаемое поведение и условия (например, `buildGroupGradesExcel_ShouldReturnResource_WhenGroupExists`, `buildStudentGradesExcel_ShouldThrow_WhenStudentNotFound`), что делает их самодокументируемыми.

Структура тестов зеркалирует структуру основного кода: для сервиса отчётов выделен класс `ReportServiceImplTest`, в котором в `@BeforeEach` подготавливаются тестовые сущности (StudyGroup, Student, Subject, User), а в тестах настраиваются моки репозиториев и проверяется результат. В модульных тестах проверяется взаимодействие с зависимостями через `Mockito.verify`: вызовы `groupRepository.findById`, `gradeRepository.findByStudentGroupGroupIdOrderByGradeDateDesc`, `studentRepository.findByGroupIdWithUser` для отчёта по группе; вызовы `studentRepository.findByIdWithUserAndGroup`, `gradeRepository.findByStudentIdOrderByDateDescWithSubject`, `attendanceRepository.findByStudentStudentIdOrderByLessonDateDescWithDetails` для отчёта по студенту; вызовы `subjectRepository.findById`, `gradeRepository.findBySubjectIdOrderByGradeDateDesc`, `attendanceRepository.findBySubjectSubjectIdOrderByLessonDateDesc` для отчёта по предмету. Проверяется также возврат непустого ресурса (`resource.exists()`), что гарантирует формирование файла (Excel или PDF).

Практические проверки включают: формирование отчёта по группе (Excel и PDF) при существующей группе и пустых списках оценок и студентов; формирование отчёта по студенту (Excel) при существующем студенте и пустых оценках и посещаемости; выброс исключения с сообщением «Студент не найден» при вызове buildStudentGradesExcel или buildStudentGradesPdf для несуществующего студента; формирование отчёта по студенту с фильтром по периоду (buildStudentGradesPdf с датами и subjectId = null); формирование отчёта по предмету (Excel и PDF) при существующем предмете. Контекстное тестирование приложения (`ServerApplicationTests`) проверяет успешную загрузку контекста Spring и запуск приложения с профилем `test`.

Интеграционные тесты при необходимости дополняют модульные: с `@SpringBootTest` и in-memory H2 можно проверить работу репозиториев и миграций (при включённом Flyway в тестах). В текущей конфигурации профиль `test` отключает Flyway и использует `ddl-auto: create-drop` для быстрого прогона тестов без поднятия внешней СУБД. Такой набор тестов даёт уверенность в ключевых сценариях формирования отчётов и обработки отсутствующих данных без стремления к стопроцентному покрытию кода; приоритет отдаётся бизнес-критичным путям и обработке ошибок.

Тесты служат не только контролем качества, но и живой документацией поведения сервиса отчётов, что облегчает поддержку и рефакторинг. Конфигурация тестов вынесена в `src/test/resources/application-test.yml` (H2 in-memory, отключённая почта, отключённый Flyway).

Результаты тестирования представлены на рисунке 1: все тесты выполняются успешно (Tests run: N, Failures: 0, Errors: 0, Skipped: 0), сборка завершается статусом BUILD SUCCESS.
